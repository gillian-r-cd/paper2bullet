# 不容辩驳的原则
* 一步一步修改，一步一步验证，一步一步更新todo
* 不要用 emoji 作为视觉标记（UI mockup 除外）
* 未经事先询问并确认，不得覆盖我的 .env 文件

# 编码模式偏好
* 始终优先选择简单的解决方案
* 尽可能避免代码重复
* 修复问题或漏洞时，在现有实现的所有方案都尝试完毕之前，不要引入新的模式或技术
* 编码时请注意合理模块化
* 在每次创建/更新文件时，都在文件最开头用注释写清楚这个文件的功能、主要函数和数据结构。

# 编程工作流偏好
* 专注于与任务相关的代码区域
* 不要修改与任务无关的代码
* 始终考虑代码更改可能影响到的其他方法和代码区域

# 保持干净状态（每次 push 前必须做到）

## Git 提交信息
* 每次 commit 必须写清楚：做了什么、为什么做。禁止使用 "wip"、"fix"、"update" 这种废话。
* 提交信息格式："[功能/模块名] 具体描述"，例如 "[auth] 增加登录失败重试限制"

## 进度记录
* 项目根目录下维护一个 progress.md 文件，每次结束编码时补充：
  - 本次做了什么
  - 还没做什么
  - 已知的问题和 bug
  - 下次开始时应该先做什么
* 这个文件是给下一次编码会话（或下一个人）看的交班记录，不是给自己看的日记。写得越具体越好。

## 功能验证
* 不要"感觉能跑"就提交。对照功能清单逐项确认。
* 如果项目有功能清单文件（feature_list.json 或类似），只有实际测试通过后才能把状态改成 passing。禁止删除或修改测试项本身。

## 回归测试
* 每次改完代码、准备提交之前，先跑一遍基础功能测试，确认没有把已有功能搞坏。
* 如果项目有 init.sh 或启动脚本，先跑起来，手动或自动走一遍核心流程。

## 代码整洁
* 提交前清理所有临时代码：调试用的 console.log、注释掉的旧代码、TODO 占位符。
* 代码要达到"可以合并到主分支"的水平：没有重大 bug，逻辑清晰，文档齐全。标准是——另一个开发者打开这个项目，能直接开始干活，不用先帮你擦屁股。

## 每次新会话开始时
* 先读 progress.md 和最近的 git log，搞清楚上次做到哪了
* 跑一遍基础测试，确认当前代码能正常工作
* 然后再开始新的开发

# 端到端测试（像用户一样测，不要像开发者一样测）
## 核心原则
* 单元测试通过 ≠ 功能可用。curl 返回 200 ≠ 用户能正常使用。
* 每个功能完成后，必须以用户视角走完整个操作流程。例如：打开浏览器 → 点击按钮 → 填写表单 → 提交 → 看到正确结果。整条链路走通才算测试通过。
* 禁止仅凭“代码层面看起来没问题”就宣布功能完成。
## 如何测
* 如果是 Web 项目，启动开发服务器，用浏览器（或浏览器自动化工具如 Puppeteer）实际操作一遍。
* 如果是 CLI 或后端项目，模拟用户的完整调用链路，而不是只测单个函数。
* 测试时必须覆盖正常路径和异常路径（比如输入为空、网络超时、未登录状态）。
## 不要相信自己的“测试通过”声明
* 如果你只跑了单元测试或 curl，明确告诉我：“我只做了 API 层测试，还没有做端到端验证”。不要把单元测试通过包装成“功能已完成”。
* 浏览器自动化工具有盲区：原生 alert 弹窗、动画效果、拖拽交互等可能截图看不出来。对这类功能，请标注“需要人工验证”而不是自行判定通过。
## 每次新会话开始时，先跑一遍核心流程
* 在开始新功能开发之前，先启动应用，走一遍最基础的用户流程（比如打开页面、登录、提交一条数据），确认应用没有处于坏掉的状态。
* 如果发现已有功能坏了，先修复再做新功能。绝对不要在坏掉的基础上继续堆代码。
